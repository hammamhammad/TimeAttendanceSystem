# TC-E2E: Cross-Module End-to-End Workflows — Test Cases

## Module Overview

These test cases validate business processes that span multiple modules, pages, and API endpoints. Each test describes a complete user journey from initiation through completion, crossing module boundaries. These tests verify that the system behaves correctly as an integrated whole, not just as isolated features.

**Modules Covered**: Authentication, Organization, Time & Attendance, Leave Management, Excuse Management, Remote Work, Approval Workflows, Payroll, Performance, Recruitment, Onboarding, Offboarding, Entitlements, Multi-Tenancy
**Applications**: Admin Portal (4200), Self-Service Portal (4201), Backend API (5099)
**Background Jobs**: DailyAttendanceGenerationJob, EndOfDayAttendanceFinalizationJob, MonthlyLeaveAccrualJob, WorkflowTimeoutProcessingJob

### A. Employee Lifecycle Flows

#### TC-E2E-001: Employee Hire-to-Work — complete onboarding to first attendance
| Field | Value |
|-------|-------|
| **Priority** | P0 |
| **Category** | Integration |
| **Pages** | Admin: /employees/create, /shifts/assign; API: DailyAttendanceGenerationJob |
| **Role** | System Admin, Employee |

**Preconditions:**
1. Active tenant with Enterprise plan (all modules)
2. Shift "Regular 08:00-17:00" exists
3. Department "IT" exists under Headquarters branch

**Steps:**
1. **Admin Portal**: Create new employee at /employees/create with branch=Headquarters, department=IT
2. **Admin Portal**: Verify employee appears in /employees list
3. **Admin Portal**: Create user account for the employee at /users/create, link to employee
4. **Admin Portal**: Assign shift "Regular 08:00-17:00" to employee at /shifts/assign
5. **Backend**: Trigger DailyAttendanceGenerationJob (or wait for 2:00 AM scheduled run)
6. **Verify DB**: AttendanceRecord created for employee for today with status=Absent (no transactions yet)
7. **Self-Service**: Login as new employee, navigate to /my-attendance
8. **Mobile/API**: Submit check-in transaction `POST /api/v1/mobile/attendance/transaction`
9. **Mobile/API**: Submit check-out transaction at end of day
10. **Backend**: EndOfDayAttendanceFinalizationJob runs
11. **Verify**: Working hours calculated correctly (check-out minus check-in minus breaks)

**Expected Results:**
- Employee created with all required fields
- User account created and linked to employee
- Shift assignment active for employee
- Attendance record generated by daily job
- Check-in and check-out transactions recorded
- Working hours calculated = (check-out - check-in) - break duration
- Status updated from Absent to Present
- Overtime calculated if hours exceed shift regular hours
