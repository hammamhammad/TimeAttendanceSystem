Feature Map

Everything the shared grid does, and the inputs that toggle each piece.

Row interaction

  • No Actions column — no pencil / eye / ellipsis buttons anywhere.
  • Double-click a row to open it — routes to [detailRoute, row.id] normally, or emits (onView) when [viewMode]="true".
  • Rows carry cursor: pointer; user-select: none so they read as clickable and the double-click doesn't leave a text selection.

Row selection

  • Leftmost checkbox column — on by default ([selectable]="true").
  • Header checkbox selects / deselects all rows on the current page (indeterminate when partial).
  • Shift-click on a row checkbox for range selection.
  • Selections persist across page / sort / filter changes (keyed by row.id in a Map).
  • "N selected · Clear" pill appears in the toolbar when > 0 rows are selected.
  • (selectionChange) emits the selected row objects; clearSelection() exposed via @ViewChild.

Sorting

  • All columns sortable by default (col.sortable !== false).
  • Click a header → sends sortBy / sortDir to the backend via ApplySort, then the page is re-sorted client-side for uniform edge-case handling.
  • Dotted nav-prop paths via sortField: 'Account.AccountName' — server walks the path by reflection.
  • Null ordering (server + client agreement): nulls always sort to the end, regardless of direction. Server emits a two-step ORDER BY CASE WHEN NULL … , col; client wraps every comparator in withNullHandling.

Filtering

  • Per-column filter icon in the header — highlighted when a filter is active.
  • Popover uses position: fixed + getBoundingClientRect() to escape the table's overflow: auto container, viewport-clamped so it never extends past the scrollbar.
  • Active filters show as chips below the toolbar with × remove and "Clear all".
  • Server-side by default (filterMode: 'client'): FilterDescriptor[] serialised as JSON in ListQueryParams.Filter, applied before CountAsync + Skip + Take so pagination reflects the globally-filtered set.
  • Operator list driven by OPERATOR_CONFIG[columnType]; value widget by WIDGET_REGISTRY[columnType]. No per-type branches in the popover.

Audit columns (auto-appended)

  • Every grid automatically appends: Owner · Created On · Created By · Modified By · Modified On.
  • Feature components should not manually re-add these — they'll duplicate.
  • Defined once in AUDIT_COLUMNS and spliced in by the columns setter.
  • Opt-out with [hideAuditColumns]="true" for grids that don't read BaseEntity-shaped rows.
  • Timestamp columns use type: 'dateTime' — formatted as Apr 6, 2026, 3:45 PM via UserDatePipe in the viewer's profile timezone.

Search & toolbar

  • Toolbar-left: search input + the "N selected · Clear" pill when rows are selected.
  • Toolbar-right: "New {entity}" button. Hidden via [showCreate]="false" (used by the six top-level activity grids, since activities are created inside the regarding entity).
  • Search is case-insensitive, sent to the backend as ListQueryParams.Search, applied as WHERE Name ILIKE ... OR Code ILIKE ....
  • Toolbar buttons use a scoped compact override (4px 10px, font 12) so the global btn-erp-sm elsewhere keeps its regular size.

Pagination & layout

  • Footer padded 8/14, page buttons 26×26.
  • Page resets to 1 whenever filters, search, or sort change.
  • Table fills remaining viewport height with scrollable body, sticky headers, and horizontal scroll when columns exceed width.
  • Compact baseline: body cells 6px 14px, line-height 1.35; header cells 8px 14px, font 11.

Column types & auto-detection

  • Visual types: text, code, link, money, date, dateTime, status, boolean (renders ✓ / —).
  • Sort / filter types (12 in total): string, number, money, date, dateTime, boolean, reference, phone, email, enum, percentage, status.
  • Auto-type-detection: columns without an explicit type / sortType / filterType peek at the first non-null cell value. typeof v === 'number' → number; typeof v === 'boolean' → boolean; otherwise string.
  • Enum / reference filters auto-populate their dropdown from the backend's /filter-options?path={dottedPath} endpoint (distinct values across all pages, not just the current one).

Key inputs

  • [columns] — column definitions (audit columns are appended automatically by the setter).
  • [selectable] (default true), [showCreate] (default true), [viewMode] (default false), [hideAuditColumns] (default false).
  • [filters] / (filterChange) — controlled filter state. Signal-backed internally so the grid works both uncontrolled and controlled.
  • [filterMode] — 'client' (default) applies client filters on the page and sends descriptors to the backend; 'server' skips client filtering.
  • Events: (onCreate), (onView), (selectionChange), (filterChange).

Confirmation Modal

Built into entity-detail. Auto-triggered for Activate / Deactivate actions; opt-in for any other action via ActionDef.confirm. All current and future entities get it free — zero per-feature configuration.

Visual Design

Confirm Action

Are you sure you want to deactivate this currency?

How It Works

  • Auto-confirm actions: 'activate' and 'deactivate' (case-insensitive) always show the modal — no confirm flag needed.
  • ActionDef.confirm: true — default message: "Are you sure you want to {label} this {entityName}?"
  • ActionDef.confirm: "Custom text" — custom message shown in the modal body.
  • Dismiss: backdrop click, Cancel button, or Escape key.
  • Confirm button: inherits the action's class (color) + icon + label.

Usage in ActionDef

actions: ActionDef[] = [ // Auto-confirmed (action name match): { label: 'Activate', icon: 'fa-check-circle', class: 'btn-erp-success', action: 'activate', condition: (e) => e?.status === 'Inactive' }, { label: 'Deactivate', icon: 'fa-ban', class: 'btn-erp-warning', action: 'deactivate', condition: (e) => e?.status === 'Active' }, // Explicit confirm with custom message: { label: 'Cancel Order', icon: 'fa-times', class: 'btn-erp-danger', action: 'cancel', confirm: 'This will permanently cancel the order. Are you sure?' }, // No confirmation (default for non-listed actions): { label: 'Create Invoice', icon: 'fa-file-invoice-dollar', class: 'btn-erp-primary', action: 'create-invoice' }, ];

Centralized Sort & Filter Architecture

Strategy-pattern utilities that the data grid relies on. One source of truth per concern: adding a new column type never touches existing comparators or predicates.

Pipeline

Server Response applyFilters() sortRows() Rendered

Mode B (entity-list): server sorts/filters globally for pagination; client re-runs both passes on the returned page for uniform edge-case handling.

File Layout

shared/utils/sorting/ ← 12-type sort engine (89 tests) types.ts ColumnType, ColumnConfig, ComparatorFn, MultiSortKey parsers.ts parseNumber, parseMoney, parsePhone, parseDate, ... with-null-handling.ts HOF: nulls always last, direction-aware, throw recovery comparators.ts COMPARATOR_REGISTRY: Record<ColumnType, ComparatorFn> sort-utils.ts sortRows(), sortRowsByColumn(), resolveFieldValue() shared/utils/filtering/ ← 12-type filter engine (72 tests) types.ts FilterOperator (22), FilterDescriptor, FilterPredicateFn with-null-handling.ts HOF: isEmpty/isNotEmpty universal, null → false operators.ts OPERATOR_CONFIG: Record<ColumnType, OperatorConfig[]> predicates/ One file per type + _numeric-base + _date-base + registry filter-utils.ts applyFilters() with memoization shared/components/filter-panel/ ← Filter UI filter-popover.component.ts Operator <select> + dynamic widget via ngComponentOutlet filter-chips-bar.component.ts Active-filter chips + "Clear all" widgets/ 5 shared components covering 12 types via WIDGET_REGISTRY src/ERP.Application/Common/Extensions/ ← Backend helpers QueryableSortExtensions.cs ApplySort<T>(query, p) — dotted-path reflection QueryableFilterExtensions.cs ApplyFilters<T>(query, p) — 22-operator Expression builder QueryableDistinctOptionsExtensions.cs GetDistinctStringsAsync<T>(query, path) — string + enum

Adding a New Column Type

  1. Add the literal to ColumnType union in sorting/types.ts.
  2. Create sorting/comparators.ts — add one core comparator + one line in COMPARATOR_REGISTRY.
  3. Create filtering/predicates/<type>.predicate.ts — one exported function.
  4. Register in PREDICATE_REGISTRY, OPERATOR_CONFIG, WIDGET_REGISTRY — one line each.

Zero changes to existing comparators, predicates, or the DataGrid component.
